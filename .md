2) Short Research Notes
What is refactoring?
Refactoring is the process of improving code structure and readability without changing its external behavior.
How is refactoring different from adding features?
Refactoring improves existing code quality, while adding features introduces new functionality.
3 code smells common in AI-generated code:
Long functions with too many responsibilities
Duplicate or repeated logic
Poor or unclear variable naming
2 techniques to refactor safely:
Make small, incremental changes with commits
Test functionality before and after each change


3) Code Quality Audit (minimum 5 findings)
Finding 1:

Issue: The controller contains business logic instead of delegating to the service.
Why: Makes the code harder to maintain and reuse long-term.
Where: user.controller.ts, inside createUser() function

Finding 2:

Issue: Duplicate validation logic is repeated in multiple functions.
Why: Increases risk of inconsistencies when updates are needed.
Where: user.service.ts, createUser() and updateUser()

Finding 3:

Issue: Function is too long and handles multiple responsibilities.
Why: Hard to read, test, and debug.
Where: user.service.ts, processUserData()

Finding 4:

Issue: Unclear variable names like data, temp, res.
Why: Reduces readability and understanding of code.
Where: Throughout both files

Finding 5:

Issue: Inconsistent error handling and missing proper status codes.
Why: Can lead to unclear API responses and debugging difficulty.
Where: user.controller.ts, error handling blocks


4) Safe Refactoring Plan (minimum 6 steps)
Step 1 (small change): Rename unclear variables
Replace data, temp with meaningful names like userData, validatedUser
Verification: Run app and test endpoints
Commit message: "refactor: improve variable naming for clarity"
Step 2: Move business logic from controller to service
Extract logic from controller into service methods
Verification: Compare API responses before and after
Commit message: "refactor: move business logic to service layer"
Step 3: Extract duplicate validation into helper function
Create reusable validation function
Verification: Test create and update endpoints
Commit message: "refactor: extract shared validation logic"
Step 4: Break large functions into smaller ones
Split processUserData() into smaller functions
Verification: Ensure same output results
Commit message: "refactor: split large function into smaller units"
Step 5: Standardize error handling
Use consistent error responses and status codes
Verification: Test error cases using curl/Postman
Commit message: "refactor: standardize error handling"
Step 6: Add basic DTO/types
Define input/output types for functions
Verification: Check TypeScript errors and run app
Commit message: "refactor: add DTOs for type safety"
Step 7: Clean up unused code
Remove commented or unused variables/functions
Verification: Run app and check no missing dependencies

5) Reflection (4–6 sentences)

AI is good at quickly generating code and providing helpful suggestions, especially for repetitive tasks or boilerplate code. However, AI is not always reliable in maintaining clean structure, as it may produce code with hidden issues or poor organization. A human engineer must always be responsible for reviewing, testing, and ensuring the code is maintainable and correct. Owning the problem means understanding the requirements deeply, thinking critically about solutions, and not just relying on generated code. It also involves making sure the system works correctly in real-world scenarios, including edge cases. Ultimately, developers must take accountability for the quality and reliability of the final product.
